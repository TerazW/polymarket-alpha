# Belief Reaction System - 用戶白皮書

> **"看存在没意义，看反应才有意义"**
> Watching existence is meaningless; watching reactions is meaningful.

---

## 1. 什麼是 Belief Reaction System？

Belief Reaction System (BRS) 是一個 **市場微觀結構證據系統**。它觀察市場中發生的事件，記錄可驗證的證據，並呈現這些證據供用戶理解市場微觀結構的當前狀態。

### 這個系統是什麼

- ✅ **證據記錄系統** - 記錄市場中可觀察到的事件
- ✅ **結構分析工具** - 分析訂單簿的結構性變化
- ✅ **反應分類引擎** - 將市場反應分類為可描述的類型
- ✅ **歷史回放平台** - 支持確定性回放以驗證證據

### 這個系統不是什麼

- ❌ **不是交易訊號系統** - 我們不提供買賣建議
- ❌ **不是預測引擎** - 我們不預測未來價格走向
- ❌ **不是自動交易工具** - 我們不執行任何交易操作
- ❌ **不是投資顧問** - 我們不提供任何投資建議

---

## 2. 核心範式：觀察反應，而非存在

傳統方法關注「市場上有什麼」—— 深度、價格、成交量。

BRS 關注「市場如何反應」—— 當深度被測試時發生了什麼。

### 為什麼反應比存在更重要？

靜態的訂單簿快照無法告訴你：
- 這些訂單是否會在壓力下保持？
- 做市商是否願意防守這個價位？
- 流動性是真實的還是會在成交前消失？

**只有觀察「測試」發生時的反應，才能獲得這些訊息。**

## 3. 系統如何運作？

### 3.1 數據收集

系統連接市場 WebSocket，以 250 毫秒的時間桶精度收集：
- **訂單簿數據** - 每個價位的深度變化
- **成交數據** - 所有成交記錄
- **價格變動** - 市場價格的變化

### 3.2 事件檢測

系統檢測三類核心事件：

#### Shock（沖擊事件）
當成交活動對某個價位造成顯著影響時觸發。
- **觸發條件**: 成交量超過基準深度的 35%，或連續 3 筆以上成交
- **記錄內容**: 時間、價位、深度基準、成交量

#### Reaction（反應事件）
Shock 發生後，系統觀察該價位的流動性如何變化。

#### Leading Event（領先事件）
不依賴成交觸發的預警信號，例如無成交的撤退。

### 3.3 反應分類

系統將觀察到的反應分類為 **7 種類型**（按優先級排序）：

| 類型 | 描述 | 觀察到的現象 |
|------|------|--------------|
| **VACUUM** | 流動性真空 | 深度驟降至接近零，且持續超過 3 秒 |
| **SWEEP** | 多檔被清 | 連續成交移除多個價位的流動性 |
| **CHASE** | 價格遷移 | 流動性僅在偏移後的價位重新出現 |
| **PULL** | 撤單退出 | 沖擊後流動性立即被取消（非成交消耗） |
| **HOLD** | 防守補單 | 流動性在限定時間窗口內恢復 |
| **DELAYED** | 延遲反應 | 流動性延遲或部分恢復 |
| **NO_IMPACT** | 無顯著影響 | 觀察到的變化未超過反應閾值 |

### 3.4 信念狀態

基於反應事件的累積，系統維護一個 **4 階段狀態機**：

| 狀態 | 指示器 | 含義 |
|------|--------|------|
| **STABLE** | 🟢 | 深度防守活躍，HOLD 反應佔主導 |
| **FRAGILE** | 🟡 | 出現壓力跡象，開始檢測到 PULL 或領先事件 |
| **CRACKING** | 🟠 | 顯著壓力，VACUUM 或 PULL 累積 |
| **BROKEN** | 🔴 | 深度防守失效，流動性結構受損 |

**重要**: 這些狀態描述的是 **可觀察的結構性變化**，不是價格方向預測。

## 4. 核心功能

### 4.1 Radar（雷達視圖）

多市場概覽，顯示每個市場的當前信念狀態和關鍵指標。

**顯示內容**:
- 信念狀態及持續時間
- 過去 10 分鐘的領先事件數量
- 脆弱性指數
- 數據健康狀態

### 4.2 Evidence Player（證據播放器）

沉浸式證據審查工具，用於深入分析特定市場。

**功能**:
- **深度熱力圖** - 可視化訂單簿隨時間的變化
- **事件時間線** - 顯示所有檢測到的事件
- **反應詳情** - 每個反應的完整證據鏈
- **狀態變化追蹤** - 顯示狀態轉換的觸發證據

### 4.3 Heatmap Tiles（熱力圖瓦片）

高保真深度可視化系統，支持多級細節（LOD）：

| LOD | 時間分辨率 | 用途 |
|-----|-----------|------|
| 250ms | 每列 250 毫秒 | 詳細分析 |
| 1s | 每列 1 秒 | 中等縮放 |
| 5s | 每列 5 秒 | 概覽模式 |

### 4.4 Alert System（告警系統）

基於檢測到的事件生成告警，按嚴重級別分類：

| 級別 | 觸發條件 |
|------|----------|
| **CRITICAL** | BROKEN 狀態、VACUUM 反應、DEPTH_COLLAPSE 領先事件 |
| **HIGH** | CRACKING 狀態、PULL 反應、PRE_SHOCK_PULL 領先事件 |
| **MEDIUM** | FRAGILE 狀態、SWEEP 反應 |
| **LOW** | 一般性 Shock 檢測 |

**重要**: 告警表示 **檢測到的事件**，不是行動建議。

### 4.5 Replay System（回放系統）

支持歷史事件的確定性回放，用於：
- 驗證系統輸出的一致性
- 研究歷史案例
- 審計檢測邏輯

**確定性保證**: 相同輸入始終產生相同輸出。

### 4.6 Evidence Chain（證據鏈）

顯示告警的完整溯源：

```
Shock → Reaction → Leading Event → State Change → Alert
```

每個節點都有時間戳和關聯證據，確保可追溯性。

### 4.7 Similar Cases（相似案例）

搜索歷史中具有相似反應模式的案例。

**重要**: 此功能 **只顯示相似的證據模式**，**不顯示後續結果**。這是設計使然，因為：
- 過去的模式不預測未來
- 我們提供對比分析的工具，不是預測工具

## 5. 數據完整性

### 5.1 Evidence Grade（證據等級）

系統對每個證據包評估數據質量：

| 等級 | 含義 |
|------|------|
| **A** | 數據完整，無缺口 |
| **B** | 輕微問題，不影響分析 |
| **C** | 存在數據缺口，結果可能受影響 |
| **D** | 數據嚴重受損，結果不可靠 |

### 5.2 Bundle Hash（證據哈希）

每個證據包都有加密哈希，用於：
- 驗證數據未被篡改
- 確保回放時數據一致性
- 支持獨立審計

### 5.3 數據健康監控

系統持續監控：
- 缺失的時間桶數量
- 重建事件計數
- 哈希校驗失敗

## 6. 技術架構

### 6.1 組件概覽

```
Polymarket WebSocket
        ↓
    Collector (數據收集)
        ↓
    Redis/TimescaleDB (存儲)
        ↓
    Reactor (事件檢測)
        ↓
    API/Tile Worker (服務)
        ↓
    Frontend (用戶界面)
```

### 6.2 時序數據存儲

使用 TimescaleDB 存儲時序數據，支持：
- 250ms 時間桶精度
- 自動數據壓縮
- 降採樣聚合（1s、1m）
- 數據保留策略

### 6.3 實時流

WebSocket API 支持實時事件推送：
- Shock 檢測
- 反應分類
- 狀態變化
- 數據健康警告

## 7. API 概覽

### 7.1 主要端點

| 端點 | 用途 |
|------|------|
| `GET /v1/radar` | 獲取多市場概覽 |
| `GET /v1/evidence` | 獲取特定市場的證據包 |
| `GET /v1/alerts` | 獲取告警列表 |
| `GET /v1/heatmap/tiles` | 獲取熱力圖瓦片 |
| `GET /v1/replay/catalog` | 獲取可回放事件目錄 |
| `WS /v1/stream` | 實時事件流 |

### 7.2 認證

API 使用 API Key 認證：
```
Authorization: Bearer brm_xxxxx
```

### 7.3 訂閱層級

| 層級 | 功能 |
|------|------|
| **Observer** | 基本 Radar 訪問，1 個 WebSocket 連接 |
| **Analyst** | 完整 Evidence 訪問，2 個連接 |
| **Institution** | 全部功能，5 個連接，批量回放 |

## 8. 使用限制與免責聲明

### 8.1 這個系統不提供

- 任何形式的交易建議
- 價格或結果預測
- 買賣時機建議
- 投資組合管理建議

### 8.2 用戶責任

- 用戶必須獨立做出所有決策
- 用戶必須理解所有風險
- 用戶不應將系統輸出視為行動指令

### 8.3 數據延遲

- 系統處理存在固有延遲
- 實時流不保證零延遲
- 歷史數據可能存在缺口

### 8.4 系統限制

- 只觀察公開市場數據
- 不能檢測所有類型的市場行為
- 分類結果基於預定義規則

### 允許的術語

| 術語 | 定義 |
|------|------|
| **Shock** | 成交活動對價位造成顯著影響的事件 |
| **Reaction** | Shock 後觀察到的流動性變化模式 |
| **VACUUM** | 流動性驟降至接近零並持續的狀態 |
| **SWEEP** | 連續成交移除多檔流動性 |
