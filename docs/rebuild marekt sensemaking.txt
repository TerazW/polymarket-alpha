必须做图，但不是传统 orderflow 图，而是**“证据可视化”**：

1. **Probability Heatmap（概率热力图）**：Bookmap 的灵魂，但我们把 x 轴变成时间、y 轴变成概率（价格），颜色是“挂单深度/驻留时间/稳定性”
2. **Reaction Timeline（反应时间线）**：每次冲击事件（hit / sweep / pull / refill）形成可回放的“行为序列”
3. **Evidence Cards（证据卡片）**：把“结论”变成可解释证据（比如：70% 有墙，但被打掉后 2 秒内回补 80%，持续 12 分钟未迁移）

## 2) 需要什么数据（最小闭环 vs 完整版）

### 最小闭环（MVP 必须）

你只需要做到：**实时 L2 orderbook + 能拿到市场元数据**

* **WebSocket Market Channel：L2 盘口**

  * `book` 消息：订阅时/交易影响盘口时推送一份聚合档位（bids/asks）
  * `price_change` 消息：有新挂单/撤单时推送“某价位的新聚合数量”等增量变化（含 best_bid/best_ask）
* **REST：补全/校验快照**

  * `GET /book?token_id=...` 拿该 token 的 orderbook summary（bids/asks、tick_size、min_order_size、neg_risk）
* **Gamma Markets API：拿市场信息（token id / condition id / tick 等）**

  * `GET /markets` 返回 `conditionId`、`clobTokenIds`、`bestBid/bestAsk`、`orderPriceMinTickSize`、`orderMinSize`、`negRisk...` 等字段 

> 结论：**做 heatmap + 做反应指标，这三类数据已经够了。**

### 完整版（提升“反应解释力”）

* **User Channel（如果你要做“我的订单/我的成交”层）**

  * user channel 需要 auth，能推送你的 trade / order 状态变化（PLACEMENT/UPDATE/CANCELLATION 等）
* **Data API / Trades（如果你要做“谁在成交推动概率”）**

  * 你给的节选里明确有 Data API `/trades` 并有独立限速 
  * 但：做“领先型反应”其实**不依赖**全量成交明细，盘口行为本身就够领先；成交更多用于“验证/注释”。

## 3) 我们做什么指标（全部围绕“反应”）

我把指标分成三层：**证据层 → 行为层 → 信号层**。你想做“leading”，核心在“行为层”。

### 3.1 证据层（Liquidity Evidence）

这些是热力图背后的“可量化证据”：

1. **Level Persistence（档位驻留）**

* 在某价位（概率）上，聚合 size 连续存在的时长（允许小幅抖动）
* 输出：P70 的 “守墙时间”、P45 的 “易碎时间”

2. **Refill Ratio（被打后的回补比）**

* 事件：该价位 size 被冲击显著下降（hit/sweep）
* 观察窗口：t+Δ（例如 1s/3s/10s）
* 指标：`refill = (size_after_window - size_after_hit) / size_before_hit`
* 意义：真假墙最关键的证据之一（你说的“看反应才有意义”）

3. **Pull Risk（撤单风险/塌陷概率）**

* 在关键节点前，深度是否出现“同步撤退”（多个价位同时下降）
* 指标：单位时间内，若多个档位 size 同时下降的比例与速度

4. **Anchor Stability（锚点稳定性）**

* 识别“信念锚点”：长时间高深度集中价位（不是 POC 那种历史统计）
* 指标：锚点价位是否迁移、迁移速度、迁移前是否先塌陷

### 3.2 行为层（Reaction Primitives）

把盘口变化拆成“可组合的行为原子”，这就是我们领先的关键：

* **STACK**：某一侧/某一价位 size 持续增加（更像“防守增强”）
* **PULL**：挂单突然消失（更像“信念撤退/信息冲击”）
* **ABSORB**：对手连续冲击但价位不让（size 下降后快速回补）
* **SWEEP**：多档被打穿（概率快速跳变）
* **REPRICE**：best_bid/best_ask 台阶式移动（情绪/信念重定价）

`price_change` 消息天然就是为这些行为服务的：它告诉你哪个价位 size 变成多少、方向 BUY/SELL 。

### 3.3 信号层（给用户的“可执行结论”）

把行为组合成你要的“产品信号”（不是交易信号，而是“信念变动提示”）：

* **“墙是真的”**：高驻留 + 高回补 + 冲击后锚点不迁移
* **“墙是锚定”**：高驻留但**低回补**，且在关键冲击/时点前会撤
* **“信息前兆”**：事件节点前出现“提前塌陷/提前迁移”，并伴随 spread/深度结构改变
* **“共识重定价进行中”**：连续 sweep + reprice，且新锚点快速形成

## 4) 做什么图（与你的理念完全一致）

1. **Liquidity Heatmap（核心）**

* y：概率（价格档位，按 tick_size 栅格化）
* x：时间
* color：聚合 size（可选对数）
* overlay：

* Top 3 “最值得注意的概率区间”

## 5) 工程实现（Web + 数据库 + 硬件）

### 5.1 能做成 Web 吗？

能，而且应该做成 Web：

* 前端：WebGL/Canvas（热力图必须高性能渲染）
* 后端：WebSocket collector + 指标计算服务
* 推送给前端：你自己的 ws（别让浏览器直接扛 Polymarket ws，后面多市场会爆）

### 5.2 数据流架构（建议）

**Collector（常驻）**

* 连接 `wss://ws-subscriptions-clob.polymarket.com/ws/market`
* 订阅一组 token_ids
* 消息类型：

  * `book`：用来初始化/纠错（快照）
  * `price_change`：用来增量更新与事件检测 

**State Store（内存态）**

* 每个 token 维护一个 L2 map：price → size（分 bid/ask）
* 每次 price_change 更新后：

  * 触发“行为原子识别”
  * 写入 timeseries（用于热力图回放/复盘）

**Timeseries DB（落地）**
两类表就够你 MVP 跑起来：

1. `book_level_ts(token_id, ts, side, price, size)`

   * 用于热力图（可降采样：比如 200ms 或 1s 聚合一次）
2. `events_ts(token_id, ts, event_type, payload)`

   * 用于 reaction tape 与证据卡片

Postgres + TimescaleDB（或 ClickHouse）都行：

* Postgres/Timescale：开发快、适合 MVP
* ClickHouse：更强吞吐，适合未来大量市场 + 高频写入

### 5.3 硬件需求（现实估算）

取决于你订阅的市场数量 M、每个市场档位深度 L、消息频率 F。

**MVP（10–50 个 token）**

* 1 台普通云机就够：2–4 vCPU / 8–16GB RAM
* Postgres：同机或托管都行

**扩展（200–1000 个 token）**

* collector 分片（按 market/tag 分）
* DB 独立（托管 Postgres/Timescale 或 ClickHouse）
* 前端要做“按需加载”（用户看哪个 market 才拉哪个时间窗的数据）

### 5.4 现成 API 限速是否会卡住？

文档里明确：Cloudflare throttling（超限会被延迟排队）
但我们主要靠 **WebSocket**，REST 只是补快照与元数据；只要你别用 REST 高频轮询，限速不是瓶颈。

## 1) 做 refill / pull / persistence 需要哪些数据？

### 必须数据 1：L2 流动性时间序列（核心）

**来源：WebSocket Market Channel**

你要持续拿到每个价位的聚合深度随时间变化（这就是“流动性数据”本体）：

* `book`：订阅时 + 发生影响 orderbook 的 trade 时，会推送聚合档位快照（bids/asks）
* `price_change`：挂单/撤单时，给出某价位新的聚合 size（“new aggregate size for price level”）+ best_bid/best_ask 
* `tick_size_change`：tick 变化事件（避免 heatmap 栅格错位）

> **结论**：只要你把 `price_change` 变成 `(token_id, side, price, size, ts)` 的连续流，就能计算 refill、pull、vacuum、persistence。

---

### 必须数据 2：冲击（shock）时间锚点

**来源：WebSocket `last_trade_price`**

* `last_trade_price`：maker/taker 撮合产生 trade 时推送（price/side/size/timestamp）

> **shock** 不需要“全量成交明细”也能做：
> 你可以用 `last_trade_price` 作为“冲击发生”的实时锚点，再观察接下来 1–20 秒内该价位的 `price_change` 回补行为。

---

### 必须数据 3：断线/漏包纠错快照

**来源：CLOB REST /book /books**

当 WS 断线、或者你的本地 orderbook 状态出现不一致，你必须能“一键拉回真相”。你需要：

* `GET /book?token_id=...`：返回该 token 的 bids/asks、tick_size、min_order_size、neg_risk 等（用于重建/校验）
* `POST /books`：一次最多 500 token 拉快照（用于批量恢复）

---

### 必须数据 4：市场映射与元数据

**来源：Gamma `/markets`**

你需要从 Gamma 拿到：

* conditionId（market）
* clobTokenIds（YES/NO token_id）
* tick/min size/bestBid/bestAsk（初始化与 sanity check）

Gamma `GET /markets` 返回字段里明确有 `conditionId`、`clobTokenIds`、`orderPriceMinTickSize`、`orderMinSize`、`bestBid`、`bestAsk` 等。

## 2) 你贴的 Data API `/trades` 在我们系统里该怎么用？

这条接口不是做 refill 的“必需品”，但它对两件事很有价值：

### 用途 A：回放与研究（补全 trade tape）

`GET /trades` 支持按 `market`（condition IDs 列表）或 `eventId` 拉成交记录，且 `user` 参数是可选的。
这意味着：你很可能能拉到“市场级 trades”（而不是只拉某个用户的 trades）。

你可以用它做：

* 事后回放（把 shock 序列做得更完整）
* 检测“连续 sweep”（多笔成交快速发生）
* 训练/校准你的 shock 阈值（例如：什么成交规模算“有效冲击”）

### 用途 B：跨市场“信息扩散”对齐

你如果监控同一 event 下多个 market，`eventId` 过滤能让你快速比对：哪个 market 的成交先爆、哪个后爆。

> 但我会提醒一个工程点：这接口是分页的（limit/offset），更适合“回放/抽样研究”，实时主力仍然是 WS。
